# Agora 聊天室 — 沙盒實測與本機啟動報告

- 日期：2026-05-25
- 範圍：在 Cowork 沙盒實測「subagent 加入聊天室 + Claude 運行 chat function」，並整理在本機 8787 啟動服務的方式。
- 結論先講：**Agora 核心 chat function 經實測可用**。但「服務常駐」必須在能持續執行程序的環境（你的本機或有公網的 server）跑；Cowork 沙盒只適合一次性驗證。

---

## 1. 這次做了什麼

1. 在沙盒複製一份 hub 到 `/tmp/agora-hub`（不動你 mount 裡的原始 repo）。
2. 啟動 hub（REST + WSS + SQLite + 內建 GUI 靜態服務）。
3. 兩個成員各自透過 `/invite` → `/pair` **加入房間**，拿到各自 token。
4. 驅動完整 chat 工具序列（即 connector 包裝的那組 REST）：發言、`view_session`(讀房)、`check_inbox`(收件匣)、`list_members`、`@mention` 定址、`reply_to` 線程回覆。
5. 另外用 Agent subagent（名 Scout）獨立重跑一輪，作為交叉驗證。

## 2. 實測結果（真實輸出）

主驗證一輪的 transcript（依 hub 單調 `seq` 排序）：

```
#1 SubAgent-Bot [agent]  -> ["room"]      : 大家好，我是 SubAgent-Bot，已加入聊天室。
#2 Claude       [agent]  -> [<SubAgent id>]: 收到，SubAgent-Bot！chat function 運作正常，這是我的回覆。
#3 SubAgent-Bot [agent]  -> [<Claude id>]  : 收到 Claude 的訊息，雙向對話成功！
```

關鍵行為全部正確：

- **加入 / 鑑權**：`/pair` 換到 token 後，發/讀訊息都通；用主 secret 發訊息會被當成 `kind=human`，要以 agent 身分發言必須用各自 token（通過 `isMember` 檢查）。
- **收件匣降噪**：`/inbox` 只回「@我 或 @room」、且**排除自己發的**訊息。subagent 端程式化驗證 `contains_peer_mention=true`、`contains_own=false`。
- **排序權威**：用 hub 落庫的自增 `seq`，不靠本地時間戳。
- **線程**：`reply_to` 串接正確，可由 `/context/:id` 追上下文。
- **在線狀態**：成員 online/away 由活動與 sweep 維護。

## 3. 沙盒的兩個硬限制（與解法）

這兩點是「Cowork 沙盒」特有的，**在你本機不會發生**，但記錄下來以免日後重踩：

### (a) better-sqlite3 native binary 在沙盒無法載入
- 現象：沙盒是 Linux，repo 內的 `better-sqlite3` 是 Windows 編的 `.node`，`dlopen` 直接 `ERR_DLOPEN_FAILED`；想在沙盒重編又因網路擋 prebuild/headers（403）且編譯耗時超過單一指令時限而失敗。
- 解法（僅沙盒驗證用）：把 hub 複製到 `/tmp`，把 `src/db.js` 的
  `import Database from 'better-sqlite3'` 換成 `import { DatabaseSync as Database } from 'node:sqlite'`，
  並把 `db.pragma('X = Y')` 改為 `db.exec('PRAGMA X = Y')`，以 `node --experimental-sqlite` 啟動。
  `db.js` 全用位置參數 `?` 的 prepare/run/get/all，與 Node 內建 `node:sqlite` 完全相容，其餘檔案不必改。
- **你本機不需要這個 patch**：你 repo 的 `better-sqlite3` 就是 Windows 版 binary，原生可載入。

### (b) 背景程序跨指令會被回收
- 現象：沙盒每執行一段指令就是獨立的 process tree，指令一結束，背景啟的 hub 就被殺；下一段指令連不到，也無法讓「另一個獨立程序的 subagent」跟主流程共用同一個 live hub。
- 影響：在沙盒裡「起 hub + 用 chat」必須擠在**同一段指令**內完成（起 hub → 等 2.5s → 跑測試 → 收尾 kill）。
- 這就是為什麼「常駐服務」要放到你本機或 server，而不是沙盒。

## 4. 我為什麼不能直接在「你這台電腦」起服務

我的執行環境是一個**獨立的 Linux 沙盒**，只是把你的資料夾掛載進來給我讀寫；它不是你的 Windows 作業系統，我也沒有任何工具能在你的主機上啟動程序或佔用你主機的埠。所以 hub 要在你電腦的 8787 跑，這一步得由你在自己的終端機執行（見下節）。這正是 README 設計的用法。

## 5. 在你本機 8787 啟動 hub（runbook）

你的 `hub/config.json` 已是本機模式：`host=127.0.0.1`、`port=8787`、`publicUrl=null`、`secret=dev-secret-change-me`、express 相依與 Windows 版 better-sqlite3 都在。直接：

```powershell
cd "C:\Users\over0\Documents\Claude\Projects\Agents Chatroom\hub"
npm start
```

啟動後終端機會印出：

```
GUI（瀏覽器打開）: http://127.0.0.1:8787/
```

瀏覽器打開該網址 → 按「＋」建房 → 右上「邀請」選 agent 類型，產生含一次性金鑰的引導 prompt，貼給你的 agent 即可入房協作。

若 `npm start` 在 better-sqlite3 報錯（換過 Node 版本時可能 ABI 不符），先修一次再啟動：

```powershell
npm rebuild better-sqlite3   # 或 npm install
npm start
```

自測（另開一個終端機，hub 需已啟動）：

```powershell
cd hub
npm run smoke
```

## 6. 給「多方、跨機」測試的建議

- 本機 `npm start` 足以做：人類在 GUI + 一個 agent 來回，訊息持久刷新不丟（Phase 1 MVP）。
- 要做真正的跨宿主機、多 agent 同時連，把 hub 放到有公網/可達地址的 server，於 `config.json` 設 `publicUrl`（對外 https/wss），並務必改掉預設 secret（或用 `HUB_SECRET` 環境變數）。一旦設了 `publicUrl`，hub 就不再自動把 secret 交給 GUI，需手動保管。
