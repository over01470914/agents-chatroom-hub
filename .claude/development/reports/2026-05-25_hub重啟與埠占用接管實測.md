# Hub 重啟與「埠占用接管」實測報告

- 日期：2026-05-25
- 主題：重啟 Agora hub，並驗證「埠被占用時能否殺掉舊服務再重啟」（[hub/src/index.js](../../../hub/src/index.js) 新增的接管邏輯，當前為未提交變更）
- 平台：Windows 10、PowerShell（操作 shell 為**一般權限**，非系統管理員）

## 起始狀態

- Hub 正常運行：`/health` 回 `{"ok":true,"name":"agora-hub","version":"0.1.0"}`
- 占用 `127.0.0.1:8787` 的 LISTENING 程序：**PID 67548**，擁有者 `DESKTOP-NQNQ94A\over0`
- 線索：以一般權限查 PID 67548 的 `Path` / `CommandLine` 皆為空 → 疑似較高完整性等級（提權）程序

## 測試與結果

### 測試 1 — 非互動防呆（不該亂殺）

以一般 shell（非 TTY）直接再跑一個 `node src/index.js`：

```
⚠️  127.0.0.1:8787 已被占用。
偵測到占用者是另一個 Agora chatroom hub 實例（agora-hub v0.1.0）。
已取消啟動，沿用既有服務即可（或設 HUB_PORT 換埠另開）。
（exit 0）
```

結論：**通過**。能正確偵測埠占用、透過 `/health` 辨識占用者是 agora-hub；因 `process.stdin.isTTY` 為假（非互動），依設計**拒絕自動殺**、安全退出，舊 hub 未受影響。

### 測試 2 — 互動接管（真正跑 kill + rebind）

接管路徑被 `!process.stdin.isTTY` 這道「需有人在場」的防呆擋住。為了讓**生產程式碼**的 `probeHub → killAllOnPort → rebind` 真正執行，用一個一次性 harness 偽造 `isTTY=true` 並餵入 `y`（只模擬人按 yes，kill/rebind 仍是原始邏輯）。

結果：偵測 + 確認 yes 後，`killAllOnPort` 反覆執行 `taskkill /F /PID 67548`，**每次都失敗**，逾時後走「放棄並提示需要管理員權限」分支。harness 退出，舊 hub（67548）仍在 listen。

### 直接 taskkill 驗證根因

```
taskkill /F /PID 67548
ERROR: The process with PID 67548 could not be terminated.
Reason: Access is denied.   (exit 1)
```

**根因確認**：舊 hub（PID 67548）是用**系統管理員權限**啟動的。一般完整性等級的程序（含 hub 內建的接管功能、我的一般權限 shell）**無法終止較高完整性的程序**，即使同一個使用者帳號。這正是接管程式碼預留的「需要系統管理員權限」失敗分支。

## 處置：提權殺掉 + 一般權限重啟

經使用者選擇「管理員權限殺掉並重啟」：

1. `Start-Process taskkill -Verb RunAs`（UAC 同意）→ 提權終止 PID 67548，**exit 0**，埠釋放。
2. 以**一般權限**重新啟動 hub。

## 最終狀態

- 新 hub：**PID 79160**，LISTENING 於 `127.0.0.1:8787`
- `/health` → `{"ok":true,"name":"agora-hub","version":"0.1.0"}`
- 新程序 `CommandLine` 一般權限即可讀出（`node src\index.js`）→ 確認為**非提權**；同完整性等級，日後接管可正常殺得動
- 資料庫 `hub/agora.db` 不受重啟影響（訊息持久）

## 結論與建議

- 接管功能的**偵測 / 辨識 / 詢問 / kill / rebind 流程本身正確**。測試 1（防呆）與測試 2（真實路徑）都按設計執行。
- **唯一限制**：被接管者的權限必須 ≤ 接管者。若舊 hub 以系統管理員啟動、而接管以一般權限啟動，`taskkill` 會 `Access is denied`，接管必然失敗。
- 建議：
  1. **一律以一般權限啟動 hub**，避免完整性等級不一致導致殺不動（本次已修正為一般權限）。
  2. 程式可考慮：偵測到 `taskkill` 因 Access denied 失敗時，在 Windows 上提示「舊 hub 為提權程序，請用系統管理員終端機重試或同意 UAC 提權」，比目前統一的「可能需要管理員權限」更精準。
  3. 非互動環境（如 CI / 自動化）下接管被防呆擋住屬預期；若需自動接管，需提供互動 TTY 或另設明確的強制旗標。
